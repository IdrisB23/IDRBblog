Organization for Dummies

Introduction to Data Structures: Arrays and Linked Lists

by Idris Babay on June 22th, 2020, 11min reads

Share now

Let's face it: organization makes life much easier. You don't want to lose those valuable holiday pictures, that's why you store them in a folder on your computer. As we're creating that directory, the file manager on the OS assigns a path to it to know where to fetch it upon request. This File Manager figure is much like your mom: nothing gets lost in the house. Only difference is, instead of shouting back the whereabouts of your blue T-shirt, it goes something like: C:\house\myBedRoom \closet\topShelf\right \notLost.toldYouSo

Data Structures are containers for data and, just like that file manager, they have built-in features for storing and retrieving data. This is the somewhat abstract black-box definition. But, don't worry! With some examples, we'll develop the right intuition for these constructs. I cannot stress enough the importance of optimization. There are 2 main factors to compare performance of structures: time required for an query (retrieve/update data) and (storage) space needed. This might not strike you as crucial, but if Google were storing some redundant information along each data object, it may take extra minutes to run a simple search with a data base of that scale.

In this post we will learn about elementary structures such as Arrays and Linked Lists. In the later articles we'll use Linked Lists to implement Stacks and Queues and have a look at more advanced data structures such as Trees, Heaps, and Hash Tables.

Arrays

An array is simply a table with one row and a fixed number n of columns.

Your aunt's shoe rack is an array of sorts... an array of pairs of shoes.

If we denote the rack with A, we can access the first pair by A[0] and generally the i-th by A[i-1]. Actually, this is what's called virtual memory, that programming languages introduced to make code more readable. Under the hood, the array entries are stored in registers in the main memory(RAM), whose addresses are in hexadecimal form. So the effective address of the first pair of shoes may be something like 0x00FF58C0: 0x00FF58C0(base address) + 0(offset), the second 0x00FF58C4...

This is great! We have our array and our virtual memory in reach, but what happens if she buys her new favorite pair and wants to add it at the first position of her collection? We'd then have to shift all existing pairs one position to the left to make enough room for the flashy shoes. In other words:

  1. A[i+1] <- A[i] for all existing pairs
  2. A[0] <- newPair
That's a lot of work! If we're adding new shoes on a regular basis, we'd be better off using a linked list.

Linked Lists

The order of elements in arrays is solely determined by the indexing. But linked lists are quite democratic, in that they allow elements to arrange themselves. There are two flavors of these lists: singly and doubly. Each element of a singly linked list has an attribute "next" that points to another element. The linked list itself has an attribute called "nil" which marks the beginning and the end of our list(or the list itself if it's empty). Illustration does better work than words:

A doubly linked list is an extension of the singly version. Its elements have an extra pointer "prev", which returns the previous element.

Now that we have basic understanding of linked lists, let's optimize our solution of the favourite shoes insertion at the head of the collection problem. If our collection was a singly linked list, we simply update the attribute "next" of the brand new shoes to point at whatever "nil" was pointing at, then we replace the "next" attrivute of "nil" with the new pair of shoes itself. And that's all it takes! Two steps instead of n+1 steps to shift an entire array of n elements.

Try solving the same problem with doubly linked list instead! One of the disadvantages of linked lists, however, is that they don't provide random access of elements. To get hold of Pair 4 in the original linked List L we write: L.nil.next.next.next.next. See how the access is linear? Where we needed only 1 step: A[i+1] to access i-th element of an array, we now need i computational steps.

Conclusion

Arrays can be pretty useful if we wanted to store static data, that we needed to check out on a regular basis but not modify. Linked Lists also come in handy when checking-in information that require regular updating is what's needed. But what structure can we rely on to maintain an efficient data base, where data is updated and checked in and out too often? By the end of this series we'll be able to answer that question.

It's also worth mentioning that arrays can be multidimensional. This type is often called a matrix. A 2-dimensional n by n array is a table with n rows and n columns. To access the element in ith row and jth column, simply invoke A[i][j]. We can extend this to even more dimensions. A[0][0][0][0][0] to access first element in a 5-dimensional matrix. Now, you have a good understanding of arrays, linked lists and know how to access a 5-dimensional matrix. Tell your friends! .
